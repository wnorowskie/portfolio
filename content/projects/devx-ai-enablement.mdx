# Summary

DevX AI Enablement is internal developer-experience work focused on making AI a dependable teammate across the software development lifecycle. As part of an AI Center of Excellence, I build and design Claude Code agents, skills, and hooks — distributed through an internal marketplace of 30+ shared skills — that help engineers move from a tracked issue to reviewed, documented, tested changes with less manual overhead.

## Problem / Motivation

Engineers spend a large share of their time on repetitive, context-heavy steps: reading a ticket, orienting in the codebase, wiring up changes, updating docs, opening a merge request, and drafting a test plan. The goal was to encode that path as reusable AI tooling so the routine parts are assisted or automated — without ever letting automation outrun trust.

## Architecture

The flagship workflow is an **orchestrator skill coordinating three sub-agents**, each owning one phase:

- **planning-agent** — analyzes the ticket and explores the repo read-only, then writes a structured implementation plan (files to touch, steps, test strategy, branch/commit conventions derived from the repo's own docs).
- **implementation-agent** — implements the approved plan on a feature branch, runs the project's tests and linters, and commits only after the developer approves a summarized change report.
- **qa-agent** — runs the full suite with coverage thresholds (≥85% overall, ≥90% on new code), walks each acceptance criterion with explicit code-and-test evidence, and performs a structured security review.

The orchestrator manages ticket lifecycle transitions, validates each phase's artifacts itself — the plan file exists and is well-formed, the branch and commit are real, the tests actually pass — and requires **explicit human approval at every gate** before proceeding. The output is a merge request labeled as AI-assisted, reviewed and merged by humans like any other change.

## Key Technical Decisions + Tradeoffs

- **Human-in-the-loop at every phase, not just the end** — plan, commit, and QA each need explicit sign-off. Slower than full autonomy, but it keeps the developer the author of the change rather than its spectator.
- **Trust but verify — in code, not prompts.** The orchestrator re-runs tests and checks git state itself instead of trusting agents' completion claims. Agent output is treated as a hypothesis until validated.
- **Deterministic hooks over prompt-based guardrails** — pre/post-tool hooks block destructive commands and sensitive-file reads, validate writes, and audit every action. Unlike prompt instructions, hooks can't be talked out of their job.
- **Composable skills over monolithic prompts** — small, reusable capabilities with declared tool allowlists are easier to test, audit, and combine, at the cost of more upfront design.

## Reliability / Performance / Security

- Integrations inherit each underlying system's RBAC through MCP — agents never have more access than the developer running them.
- Per-skill tool allowlists constrain what each skill can touch; secret scanning is a hard stop that halts all other work.
- OpenTelemetry traces every session — usage, acceptance, and cost metrics flow into org dashboards, so adoption and behavior are observable rather than anecdotal.

## Results / Impact

- Roughly **3 in 4 product engineers weekly-active** on the tooling, with about half qualifying as power users; adoption spans nearly every squad.
- Tens of thousands of agent sessions with a **~98% suggestion-acceptance rate** over a trailing 30-day window.
- In org-wide surveys, a majority of engineers report saving **3+ hours per week**; over a third report 5+.
- A repeatable ticket-to-MR path that turns one-off prompting into shared, governed infrastructure.

## How Agent Output Quality Is Evaluated

- **Automatic:** guardrail hooks on every tool call; CI enforcing test metadata and a regression rule (every bug fix ships a failing-before test); structured change-summary rubrics per phase.
- **Human:** mandatory MR review with structured narratives that direct reviewer attention; plan-level review gates before any code is written; feedback skills that encode lessons from bad runs back into agent docs.
- **Longitudinal:** per-agent health scorecards in design — MR rejection rate, CI failure rate, human-override frequency, coverage deltas — reviewed on a regular cadence.

## What I'd Improve Next

- Consolidate ticket-to-MR cycle-time metrics into a canonical before/after dashboard.
- Graduate agent health scorecards and CI quality gates from pilot to universally enforced.
